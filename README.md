# AI Voice Assistant

1. 概要
本プロジェクトは、「身近な人と話しているような感覚で、作業をしながら自然に対話ができるAIアシスタント」の実現を目的としたシステムです。
ユーザーの音声入力をリアルタイムで認識し、AIがコンテキストや感情に応じた最適な回答を音声で出力する基本の会話ループを提供します。

**主要機能**
* **リアルタイム音声対話**: マイクからの音声入力をトリガーとした、遅延の少ないAIとの音声会話。
* **音声生体認証**: 登録された話者の声を識別し、特定のユーザーのみを受け付ける認証機構。
* **感情モデルの搭載**: 会話内容に応じてAIの感情状態が動的に変化し、JSONファイルへの永続化に対応。
* **長期記憶（RAG）**: ベクトルデータベースを活用し、過去の会話履歴をコンテキストとして保持・呼び出し。

---

## 2. 技術スタック
* **LLM API**: Google Gemini API (`google-genai`)
* **音声認識 (STT)**: `faster-whisper`
* **音声合成 (TTS)**: `VOICEVOX`
* **ベクトルデータベース**: `ChromaDB`
* **埋め込みモデル**: `sentence-transformers`
* **話者認識 (Biometrics)**: `SpeechBrain`
* **オーディオI/O**: `PyAudio`, `SpeechRecognition`, `winsound`

## 3. ファイル構成と役割（アーキテクチャ）

本システムは「関心の分離」を意識し、機能ごとにモジュールを分割して保守性を高めています。

| ファイル名 | 役割・概要 |
| :--- | :--- |
| `main.py` | **システムの司令塔。** 音声入力、認証、感情更新、LLM送信、音声合成の一連のメインループを制御。 |
| `config.py` | **設定値の一元管理。** 環境変数（`.env`）とは明確に区別し、ファイルパス、しきい値、感情の初期値リストなど、プログラムの挙動を決める定数を集約。 |
| `audio_handlers.py` | **音声入出力の制御。** `faster-whisper` を用いたノイズ除去付きの音声認識と、`VOICEVOX` を利用したキャラクターボイスによる音声出力処理。 |
| `auth_systems.py` | **セキュリティ・認証管理。** 話者認識（生体認証）を実行。認証精度（しきい値）を判断するためのプロファイルファイル（5パターン）の作成、登録時の録音処理を担当。 |
| `memory_manager.py` | **記憶・コンテキスト管理。** ユーザーの発言、AIの回答、それらに伴う感情の変化を追跡。ベクトルDB（ChromaDB）と連携し、過去の記憶の保存および関連情報の呼び戻しを実行。 |
| `logic_entities.py` | **共通ロジック・制限管理。** APIの過剰リクエストによる `429 Too Many Requests` エラーを防止するための、タイムスタンプを用いた実行間隔制御などを実装。 |

## 4. セットアップと環境構築
前提条件
**Pythonの推奨バージョン**: 3.12.7
**VOICEVOX**: 実行環境で事前に起動している必要があります。

手順
4.1 リポジトリのクローンとインストール
   ```bash
      git clone https://github.com/sazirosa891t5k-jpg/AI-Voice-Assistant-
      cd AI-Voice-Assistants-
      pip install -r requirements.txt
   
     .env.example をコピーして .env を作成し、必要なAPIキー（Gemini API)を設定してください。
      cp .env.example .env
     .env ファイルをエディタで開き、APIキーを入力

4.2　実行
   VOICEVOXを起動した状態で、以下のコマンドを実行してください。
   python main.py

5. 技術的なこだわり・設計思想
5.1「ウェイクワードレス」による自然な対話とトレードオフの制御
従来のAIアシスタントに必須であったウェイクワード（「Hey Siri」等）は、人間らしい会話のテンポを阻害する要因になると考えました。
本システムでは、あえて日常会話のスムーズさを最優先し、声紋認証のみで稼働するウェイクワードレス設計を採用しました。
また、セキュリティの観点からは、生体認証単体での運用はリプレイアタック等のリスクを伴うため、本来は多要素認証が望ましいのですが、コンセプトの実現のために割あえて単一認証にするアプローチを行っています。

5.2認証とデータクレンジングの「多層防御」
「声紋認証を導入しているならノイズは手前で弾かれるため、音声認識側の対策は不要ではないか」という問いに対し、本システムでは「Who（誰が）」と「What（何を）」を切り分けた多層防御ロジックを組んでいます。
Whoの防御（auth_systems.py）: 話者を識別するが、完全な無音や環境ノイズに対してモデルが誤作動を起こし、ゲートを突破させてしまうリスクに対応。
Whatの防御（audio_handlers.py）: faster-whisper 特有の、無音やノイズを学習データのバイアスによって「ご視聴ありがとうございました」等の定型文に誤認識する「無音ハルシネーション現象」を、入力フィルタリングと出力テキストクリーニングの2段階で抑制。
一度ゲートが開いた後に紛れ込む環境ノイズによるLLMの暴走を防ぐため、この二重のパイプラインを必須として実装しています。

◆ APIレートリミット（429エラー）の自主規制ロジック
短時間に連続してリクエストが送信される会話型アプリにおいて、サーバー側のリソースを枯渇させる 429 Too Many Requests エラーの回避は実務上必須です。
本システムでは、LLMへの送信直前にタイムスタンプを検証し、前回のアクセスから一定時間が経過していない場合は自動で待機（またはスキップ）する制御ロジックを組み込み、アプリケーションの堅牢性を担保しています。

◆ 感情状態の永続化
プログラムの終了によってリセットされてしまう揮発性の変数を排除し、MemoryManager（JSONファイルへの書き出し・同期）を介して起動時に過去の感情データを復元、会話の終了時に上書き保存する永続化機構を実装しています。
